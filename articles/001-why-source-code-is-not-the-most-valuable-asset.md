# สิ่งที่มีค่าที่สุดในระบบ MES ของโรงงาน ไม่ใช่ Source Code

## บทนำ

ตลอด 17 ปีที่ผ่านมา ผมสร้างระบบมากกว่า 200 กรณีใช้งานในโรงงาน ตั้งแต่การเก็บข้อมูลผลผลิต คุณภาพ พลังงาน ซ่อมบำรุง ไปจนถึงระบบบริหารบุคลากร

หลายปีต่อมาผมจึงพบว่าสิ่งที่ทำมาตลอดนั้น มีชื่อเรียกอยู่แล้วในอุตสาหกรรม นั่นคือ Manufacturing Execution System หรือ MES

หลายคนมองว่าระบบ MES คือ Software ทั่วไปที่ใช้ในองค์กร เมื่อพูดถึงการพัฒนาระบบ สิ่งแรกที่นึกถึงมักเป็น

* ภาษาโปรแกรม
* Framework
* Database
* Dashboard
* Source Code

แต่สำหรับผม การเดินทางไม่ได้เริ่มต้นจาก Software

ผมจบมาทางด้านวิศวกรรมอุตสาหการ (Industrial Engineering : IE) และเมื่อเริ่มทำงานในโรงงาน สิ่งที่ผมเชื่อมาตลอดคือ "ข้อมูล" คือทรัพยากรที่มีค่าที่สุดขององค์กร

ความเชื่อนี้ทำให้ผมเริ่มสร้างเครื่องมือสำหรับเก็บข้อมูลต่าง ๆ ภายในโรงงาน เพื่อนำข้อมูลเหล่านั้นกลับมาใช้ในการตัดสินใจ ปรับปรุงกระบวนการ และเพิ่มประสิทธิภาพการทำงาน

ในช่วงเวลานั้น ผมคิดว่าข้อมูลคือคำตอบของทุกอย่าง

แต่เมื่อเวลาผ่านมาจนถึงวันนี้ ผมพบว่าความจริงอาจไม่ได้เป็นเช่นนั้น

หลังจากได้มีโอกาสพัฒนาและอยู่กับข้อมูล อยู่กับผู้ใช้กว่า 2,000 คนในโรงงาน และมีองค์กรย่อยมากกว่า 30 องค์กร ผมเริ่มเห็นว่าหลายระบบมีความเชื่อมโยงกันอย่างน่าสนใจ

และสิ่งที่มีค่าที่สุดอาจไม่ใช่ข้อมูล ไม่ใช่โปรแกรม และไม่ใช่ Source Code อีกต่อไป

---

# 1. Source Code สามารถเขียนใหม่ได้

ตลอดระยะเวลาหลายปีที่ผ่านมา เทคโนโลยีที่ใช้พัฒนาระบบเปลี่ยนแปลงอยู่ตลอดเวลา

ในช่วงแรก ผมเริ่มจากการเก็บข้อมูลกระบวนการผลิต คุณภาพ และผลผลิตใน Excel พร้อมกับใช้ Macro และ Pivot Table ในการวิเคราะห์ข้อมูล

เมื่อปริมาณข้อมูลเพิ่มขึ้น Excel เริ่มไม่ตอบโจทย์ จึงขยับมาใช้ Microsoft Access เพื่อช่วยจัดการข้อมูลให้เป็นระบบมากขึ้น

แต่ปัญหาสำคัญยังคงอยู่

ข้อมูลในโรงงานเปลี่ยนแปลงตลอดเวลา และหลายกรณีจำเป็นต้องเห็นข้อมูลแบบ Real-time

นั่นทำให้ผมเริ่มพัฒนาระบบ Web Application และเลือกใช้ PHP เป็นเครื่องมือหลักในการพัฒนา

ตลอดหลายปีที่ผ่านมา PHP เปลี่ยนผ่านจาก Version 5 มาจนถึง Version 8 ในปัจจุบัน Framework ต่าง ๆ เกิดขึ้นและหายไป เทคโนโลยีใหม่เข้ามาแทนที่เทคโนโลยีเดิมอยู่เสมอ

บางโมดูลที่ผมพัฒนาเมื่อหลายปีก่อนยังคงทำงานอยู่ ในขณะที่บางโมดูลถูกเขียนใหม่ทั้งหมดเพื่อรองรับเทคโนโลยีรุ่นใหม่

> ไม่ใช่เพราะ Business Process เปลี่ยน
> แต่เป็นเพราะเทคโนโลยีเปลี่ยน

ในด้าน User Interface ก็เช่นเดียวกัน

ผมเคยใช้ jQuery Mobile เพื่อรองรับคอมพิวเตอร์หน้างานและ Tablet ซึ่งในยุคนั้นถือว่าใช้งานได้ดีมาก

แต่เมื่อเวลาผ่านไป ความคาดหวังของผู้ใช้งานเปลี่ยนไป

ผู้ใช้ต้องการหน้าจอที่เร็วขึ้น ใช้งานง่ายขึ้น รองรับอุปกรณ์ที่หลากหลายขึ้น และมีประสบการณ์ใช้งานที่ดีกว่าเดิม

สุดท้าย UI จึงต้องเปลี่ยน Framework จึงต้องเปลี่ยน และเทคโนโลยีก็ถูกบังคับให้เปลี่ยนตาม

แต่เมื่อมองย้อนกลับไป ผมพบเห็นสิ่งที่น่าสนใจ

ถึงแม้โปรแกรมจะถูกเขียนใหม่หลายครั้ง

ถึงแม้เทคโนโลยีจะเปลี่ยนจาก Excel ไปเป็น Access และจาก Access ไปเป็น Web Application

ถึงแม้ Framework จะถูกเปลี่ยนอยู่เรื่อย ๆ

แต่บางสิ่งกลับยังคงอยู่เสมอ

สิ่งนั้นไม่ใช่ Source Code และไม่ใช่ฐานข้อมูล

แต่เป็น "วิธีคิดในการแก้ปัญหาขององค์กร" ที่ค่อย ๆ ถูกสะสมอยู่ภายในระบบ

ทุกครั้งที่เกิดปัญหาใหม่

ทุกครั้งที่มีการแก้ไข Workflow

ทุกครั้งที่มีการเพิ่ม Validation Rule

ทุกครั้งที่มีการปรับปรุงกระบวนการทำงาน

องค์กรกำลังฝากองค์ความรู้บางอย่างไว้ภายในระบบโดยไม่รู้ตัว

และนั่นคือจุดที่ทำให้ผมเริ่มตั้งคำถามว่า

ถ้า Source Code สามารถเขียนใหม่ได้

แล้วอะไรคือสิ่งที่มีค่าจริง ๆ ที่เรากำลังสะสมอยู่ภายในระบบ MES เหล่านี้

# 2. สิ่งที่ยังคงอยู่หลังจากเปลี่ยนเทคโนโลยีหลายครั้ง

เมื่อเวลาผ่านไป โปรแกรมเปลี่ยนไปหลายเวอร์ชัน ผู้ใช้งานก็เปลี่ยนผ่านจากคนรุ่นหนึ่งไปสู่อีกรุ่นหนึ่ง

จากวันที่ผมพัฒนาระบบคนเดียว จนกลายเป็นทีมพัฒนาหลายคน และมีการแตกแขนงโมดูลออกไปมากมาย

Workflow หลายอย่างอาจถูกปรับปรุงให้ดีขึ้น

หน้าจออาจถูกออกแบบใหม่

ฐานข้อมูลอาจถูกปรับโครงสร้าง

แม้กระทั่งโปรแกรมบางตัวอาจถูกเขียนใหม่ทั้งหมด

แต่สิ่งหนึ่งที่แทบไม่เคยเปลี่ยนเลยคือ

**Business Rule**

และกลุ่มโปรแกรมที่เป็นแก่นหลักของงานนั้นจริง ๆ

ผมเรียกสิ่งเหล่านี้ว่า **Core Module**

หลังจากพัฒนาระบบมาหลายปี ผมเริ่มพบรูปแบบบางอย่าง

ไม่ว่าผมจะเปลี่ยนภาษาโปรแกรมกี่ครั้ง

ไม่ว่าหน้าจอจะถูกออกแบบใหม่กี่รอบ

สิ่งที่ยังคงอยู่เสมอไม่ใช่ Source Code

แต่เป็นองค์ความรู้ที่องค์กรใช้งานอยู่ทุกวัน

---

## 2.1 Business Rule คืออะไร

ก่อนจะพูดถึง Core Module ผมอยากอธิบาย Business Rule ก่อน

เพราะในความเป็นจริงแล้ว Business Rule คือสิ่งที่องค์กรใช้ตัดสินใจในแต่ละวัน

ตัวอย่างเช่น

* OT เกิน 36 ชั่วโมงต่อสัปดาห์ต้องแจ้งเตือนว่าผิดกฎหมาย
* ใบแจ้งซ่อมบางประเภทต้องได้รับการอนุมัติก่อนเริ่มงาน
* พนักงานระดับ Supervisor สามารถอนุมัติเอกสารได้เฉพาะหน่วยงานของตนเอง
* เครื่องจักรบางประเภทต้อง PM ทุก 30 วัน
* สินค้าบางกลุ่มต้องผ่านการตรวจคุณภาพก่อนจัดส่ง

จะเห็นว่า Business Rule ไม่ได้เป็นข้อมูล

แต่เป็น "กติกา" และ "วิธีคิด" ที่องค์กรใช้ดำเนินงาน

หลายครั้ง Business Rule เหล่านี้ไม่ได้ถูกเขียนอยู่ในคู่มือ

แต่ถูกถ่ายทอดจากรุ่นพี่สู่รุ่นน้อง หรือถูกเรียนรู้จากปัญหาที่เคยเกิดขึ้นในอดีต

และเมื่อมีการนำกฎเหล่านี้มาสร้างเป็นระบบ องค์ความรู้เหล่านั้นก็เริ่มถูกเก็บสะสมอยู่ภายใน MES

---

## 2.2 Core Module คืออะไร

ในขณะที่ Business Rule คือกติกา

Core Module เปรียบเสมือนโครงสร้างพื้นฐานของระบบ

หากมองในมุมของการพัฒนาระบบ Core Module คือกลุ่มโมดูลที่ทำหน้าที่เก็บและบริหารข้อมูลหลักที่ระบบอื่นต้องใช้งานร่วมกัน

ตัวอย่างง่ายที่สุดคือ

หากเราต้องการสร้างระบบสำหรับบริหารโรงงาน ไม่ว่าจะเป็นระบบคุณภาพ ระบบซ่อมบำรุง หรือระบบการผลิต

สิ่งแรกที่เราต้องมีคือระบบจัดการบุคลากร

เพราะทุกกิจกรรมในโรงงานล้วนเกี่ยวข้องกับคน

เราต้องรู้ว่า

* ใครคือผู้ใช้งาน
* ใครมีสิทธิ์ทำอะไร
* ใครสังกัดหน่วยงานใด
* ใครเป็นผู้อนุมัติ
* ใครเป็นผู้รับผิดชอบงาน

จากจุดเริ่มต้นเล็ก ๆ อย่างตารางผู้ใช้งานเพียงตารางเดียว

ระบบจะค่อย ๆ เติบโตเป็น

* ระบบจัดการสิทธิ์
* ระบบจัดการหน่วยงาน
* ระบบจัดการตำแหน่งงาน
* ระบบจัดการสายการบังคับบัญชา

และกลายเป็น Core Module ด้านการบริหารบุคลากรในที่สุด

เมื่อมี Core Module ด้านบุคลากรแล้ว เราจึงสามารถต่อยอดไปยังระบบอื่นได้

ยกตัวอย่างเช่น ระบบซ่อมบำรุงเครื่องจักร

หากต้องการสร้างระบบใบแจ้งซ่อมหรือ Work Order ที่สมบูรณ์ สิ่งที่ต้องมีไม่ใช่แค่หน้าจอสำหรับเปิดใบงาน

แต่ต้องมีองค์ประกอบพื้นฐานรองรับอยู่ก่อน เช่น

* ระบบจัดการเครื่องจักร
* ระบบจัดการ Functional Location
* ระบบจัดการอะไหล่
* ระบบจัดการประเภทงานซ่อม
* ระบบจัดการทีมช่าง
* ระบบจัดการผู้รับผิดชอบ

ซึ่งหลายโมดูลในกลุ่มนี้ยังต้องอาศัยข้อมูลจาก Core Module ด้านบุคลากรเข้ามาเชื่อมต่ออีกชั้นหนึ่ง

---

## 2.3 เมื่อ Business Rule และ Core Module ทำงานร่วมกัน

หากมีเพียง Core Module ระบบก็เป็นเพียงฐานข้อมูลที่เก็บข้อมูลได้เป็นระเบียบ

หากมีเพียง Business Rule แต่ไม่มี Core Module รองรับ กฎเหล่านั้นก็ไม่สามารถนำมาใช้งานจริงได้

คุณค่าที่แท้จริงของระบบ MES จึงไม่ได้เกิดจากการมีข้อมูลจำนวนมาก หรือมีหน้าจอสวยงามเพียงอย่างเดียว

แต่เกิดจากการที่ Core Module และ Business Rule ถูกเชื่อมเข้าด้วยกัน และถูกปรับปรุงอย่างต่อเนื่องจากประสบการณ์การทำงานจริง

ยิ่งระบบถูกใช้งานนานเท่าไร

ยิ่งมีการแก้ปัญหามากเท่าไร

Business Rule ก็ยิ่งเพิ่มขึ้น

Core Module ก็ยิ่งสมบูรณ์ขึ้น

และองค์ความรู้ขององค์กรก็ยิ่งถูกสะสมอยู่ภายในระบบมากขึ้นเท่านั้น

ตรงนี้เองที่ทำให้ผมเริ่มมองว่า MES ไม่ได้เป็นเพียง Software

แต่มันคือคลังความรู้ขององค์กรที่ค่อย ๆ เติบโตไปพร้อมกับธุรกิจ

# บทสรุป

เมื่อมองย้อนกลับไปตลอด 17 ปีที่ผ่านมา ผมพบว่าระบบจำนวนมากที่เคยพัฒนา บางระบบยังคงถูกใช้งานอยู่ บางระบบถูกปรับปรุงใหม่ และบางระบบถูกเขียนขึ้นใหม่ทั้งหมด

แต่สิ่งที่ยังคงอยู่เสมอ ไม่ใช่ภาษาโปรแกรม ไม่ใช่ Framework และไม่ใช่ Source Code

สิ่งที่ยังคงอยู่คือองค์ความรู้ที่ถูกสะสมผ่านการทำงานจริงขององค์กร

ไม่ว่าจะเป็น

* วิธีการทำงาน
* ขั้นตอนการอนุมัติ
* กฎทางธุรกิจ
* ข้อจำกัดของกระบวนการ
* ประสบการณ์จากปัญหาที่เคยเกิดขึ้น
* แนวทางการตัดสินใจที่ถูกปรับปรุงซ้ำแล้วซ้ำอีก

สิ่งเหล่านี้ค่อย ๆ ถูกบันทึกลงในระบบทีละน้อย จนกลายเป็นทรัพย์สินที่มีค่าที่สุดขององค์กรโดยที่หลายคนอาจไม่ทันสังเกต

วันนี้ผมจึงเริ่มมองระบบ MES แตกต่างจากเมื่อ 17 ปีก่อน

ในอดีตผมคิดว่าระบบมีหน้าที่เก็บข้อมูล

แต่วันนี้ผมมองว่าระบบมีหน้าที่เก็บ "องค์ความรู้"

และเมื่อองค์ความรู้เหล่านี้ถูกสะสมมากขึ้นเรื่อย ๆ ระบบก็เริ่มมีคุณค่ามากกว่าตัวโปรแกรมที่ใช้สร้างมันขึ้นมา

บางทีสิ่งที่มีค่าที่สุดในระบบ MES อาจไม่ใช่ Source Code

แต่อาจเป็นประสบการณ์ วิธีคิด และบทเรียนที่องค์กรได้ฝากเอาไว้ภายในระบบตลอดหลายปีที่ผ่านมา

บทความนี้เป็นเพียงจุดเริ่มต้นของสิ่งที่ผมเรียกว่า Manufacturing Use Case Library

ตลอดหลายปีที่ผ่านมา ผมมีโอกาสพัฒนาระบบในหลากหลายด้าน ไม่ว่าจะเป็น HR, Production, Maintenance, Quality, Utility & Energy, Safety รวมถึงระบบสนับสนุนการตัดสินใจในอีกหลายกระบวนการ

ในช่วงแรก ผมมองว่าระบบเหล่านี้เป็นเพียงเครื่องมือสำหรับแก้ปัญหาเฉพาะเรื่อง

แต่เมื่อเวลาผ่านไป ผมเริ่มพบว่าภายในระบบเหล่านี้มีบางสิ่งที่กำลังถูกสะสมเพิ่มขึ้นทุกวัน

นั่นคือองค์ความรู้ขององค์กร

* องค์ความรู้ที่เกิดจากการทำงานจริง
* องค์ความรู้ที่เกิดจากความผิดพลาด
* องค์ความรู้ที่เกิดจากการแก้ปัญหาซ้ำแล้วซ้ำอีก
* องค์ความรู้ที่เกิดจากประสบการณ์ของผู้คนหลายรุ่น

และองค์ความรู้เหล่านั้นค่อย ๆ ถูกถ่ายทอดลงไปอยู่ใน Core Module และ Business Rule ของแต่ละระบบโดยไม่รู้ตัว

ในอดีต การถ่ายทอดความรู้เหล่านี้อาจต้องอาศัยประสบการณ์หลายปี ต้องอาศัยการทำงานร่วมกับผู้เชี่ยวชาญ หรืออาจต้องรอให้คนรุ่นก่อนค่อย ๆ สอนคนรุ่นใหม่

แต่วันนี้ เทคโนโลยีกำลังเปลี่ยนไปอีกครั้ง

เมื่อ AI เริ่มเข้ามามีบทบาทมากขึ้น สิ่งที่น่าสนใจอาจไม่ใช่ความสามารถของ AI เพียงอย่างเดียว

แต่คือการที่เราสามารถนำองค์ความรู้ที่สะสมอยู่ภายในระบบตลอดหลายปีที่ผ่านมา ออกมาถ่ายทอด เรียนรู้ และต่อยอดได้รวดเร็วกว่าที่เคย

ในอดีต เราอาจเรียกสิ่งเหล่านี้ว่าโปรแกรม หรือระบบงาน

แต่ในมุมมองของผมวันนี้ มันคือองค์ความรู้ขององค์กรที่ถูกจัดเก็บในรูปแบบที่คอมพิวเตอร์สามารถเข้าใจได้

และบางที นี่อาจเป็นทรัพย์สินที่มีค่าที่สุดของโรงงานในยุค AI
